## Summary

This RFC defines the Forge Shell configuration model — the `config.toml` file
that controls shell behaviour, environment setup, plugin loading, prompt
rendering, key bindings, aliases, and PATH management. Configuration is
declarative, platform-aware, and version-controlled friendly. It replaces
`.bashrc`, `.zshrc`, and Oh My Zsh configuration with a single, well-specified
file.

---

## Motivation

bash and zsh configuration is a patchwork of `.bashrc`, `.bash_profile`,
`.zshrc`, `.zprofile`, `.profile`, Oh My Zsh plugins, Prezto modules, and
sourced files. The loading order is confusing, platform-specific, and
poorly documented. New developers spend hours debugging why their config
doesn't load correctly.

Forge Shell replaces this with a single `config.toml` — declarative, typed,
and with a well-defined loading order. The format is chosen for human
readability and version control friendliness.

## Design

### 1. File Location

| Platform | Path |
|---|---|
| Linux | `~/.config/forge/config.toml` |
| macOS | `~/.config/forge/config.toml` |
| Windows | `%APPDATA%\forge\config.toml` |

A project-level config can override user config for specific directories:

```
~/.config/forge/config.toml    # user config — always loaded
./.forge/config.toml           # project config — loaded if trusted
```

Project config requires `forge env trust` — same trust model as `.forge-env`.

### 2. Full `config.toml` Specification

```toml
# ─────────────────────────────────────────────
# Forge Shell Configuration
# Generated by: forge migrate / forge init
# Documentation: forge-shell.dev/docs/config
# ─────────────────────────────────────────────

# ── Environment Variables ────────────────────
[env]
EDITOR    = "nvim"
GOPATH    = "~/go"
JAVA_HOME = "/usr/lib/jvm/java-17"
FORGE_ENV = "development"

# Platform-specific env vars
[env.macos]
HOMEBREW_PREFIX = "/opt/homebrew"

[env.linux]
PACKAGE_MANAGER = "apt"

[env.windows]
USERPROFILE = "%USERPROFILE%"

# ── PATH Management ───────────────────────────
[path]
prepend = [
    "~/.local/bin",
    "/usr/local/go/bin",
    "~/.cargo/bin",
]
append = []

# Platform-specific PATH entries
[path.macos]
prepend = ["/opt/homebrew/bin"]

[path.windows]
prepend = ["C:/Users/Ajitem/AppData/Local/Programs/"]

# ── Aliases ───────────────────────────────────
[aliases]
ll  = "ls --show_hidden --long"
la  = "ls --show_hidden"
gs  = "git status"
gp  = "git push"
gl  = "git pull"
k   = "kubectl"
d   = "docker"

# ── Plugins ───────────────────────────────────
[plugins]
installed = [
    "forge-git",
    "forge-docker",
    "forge-kubectl",
    "forge-node",
    "forge-python",
]

# Local plugins — loaded from directory
local = ["~/.config/forge/plugins/"]

# ── Functions ─────────────────────────────────
[functions]
# .fgs files in this directory are auto-loaded as shell functions
autoload = "~/.config/forge/functions/"

# ── Prompt ────────────────────────────────────
[prompt]
theme    = "default"       # "default" | "minimal" | plugin theme name
style    = "powerline"     # "powerline" | "plain" | "minimal"
segments = [
    "cwd",
    "git",
    "execution_time",
    "exit_code",
]

# ── REPL ──────────────────────────────────────
[repl]
autosuggestions     = true
syntax_highlighting = true
fuzzy_completion    = true
inline_errors       = true
history_size        = 10000

# ── Key Bindings ──────────────────────────────
[keybindings]
"ctrl+r" = "history_search"
"ctrl+a" = "line_start"
"ctrl+e" = "line_end"
"ctrl+w" = "delete_word"
"ctrl+l" = "clear_screen"
"ctrl+c" = "cancel"
"ctrl+d" = "exit"
"→"      = "accept_suggestion"
"ctrl+f" = "accept_suggestion_word"

# ── Built-in Overrides ────────────────────────
[overrides]
# Override built-in commands with system versions or specific binaries
# Applies to interactive shell only — scripts are hermetic
ls   = "system"                 # use system ls
grep = "/usr/local/bin/rg"      # use ripgrep

# ── Completions ───────────────────────────────
[completions]
# Plugin-provided completions are loaded automatically from [plugins]
# Additional completion scripts can be specified here
extra = []

# ── Updates ───────────────────────────────────
[updates]
auto    = true
channel = "stable"    # "stable" | "nightly"

# ── Formatting ────────────────────────────────
[fmt]
argument_style = "named"    # "named" | "flags" | "preserve"

# ── Includes ──────────────────────────────────
[includes]
files    = []
optional = ["~/.config/forge/local.toml"]   # loaded if exists, no error if missing

# ── .env Auto-load ────────────────────────────
[env.dotenv]
autoload = false    # never auto-load .env without explicit trust
```

### 3. Platform Sections

Every top-level section supports platform subsections:

```toml
[env]
EDITOR = "nvim"        # all platforms

[env.macos]
EDITOR = "nano"        # macOS only — overrides above

[env.linux]
EDITOR = "vim"         # Linux only

[env.windows]
EDITOR = "notepad"     # Windows only
```

Platform sections are merged — the platform-specific section overrides the
base section for the current platform. Other platforms' sections are ignored.

### 4. Loading Order

```
1. User config          ~/.config/forge/config.toml
2. Includes             files listed in [includes]
3. Optional includes    files listed in [includes.optional] — if they exist
4. Project config       ./.forge/config.toml — if trusted
5. .forge-env script    ./.forge-env — if trusted
```

Later entries override earlier ones for env vars and PATH. For lists
(plugins, aliases) — they are merged, not overridden.

### 5. `forge config` CLI

```bash
forge config get env.EDITOR          # print a config value
forge config set env.EDITOR nvim     # set a config value
forge config list                    # print full resolved config
forge config validate                # validate config.toml syntax
forge config init                    # generate a default config.toml
forge config migrate                 # migrate from .bashrc/.zshrc
```

### 6. Trust Model

Project-level configs and `.forge-env` scripts require explicit trust:

```bash
forge env trust              # trust .forge-env + .forge/config.toml in cwd
forge env trust --list       # list all trusted directories
forge env trust --revoke     # revoke trust for cwd
```

Trust grants stored in `~/.config/forge/trusted-envs.toml`.

### 7. Version Control Recommendations

```gitignore
# Generated by forge init
.forge-env.local
.forge/config.local.toml
```

`config.toml` itself should be committed — it's declarative and contains no
secrets. Secrets belong in `.env` files (gitignored) loaded via
`forge::env::load()`.

---

## Drawbacks

- **TOML has limitations for complex logic.** Platform-specific behaviour
  beyond simple value overrides requires the `.forge-env` script escape hatch.
  This is intentional — config is declarative, logic belongs in scripts.
- **Single config file replaces a familiar ecosystem.** Oh My Zsh users have
  years of muscle memory. The migration path via `forge migrate` mitigates
  this but the learning curve is real.

---

## Alternatives Considered

### Alternative A — Use `.fgs` file as config

**Rejected:** A ForgeScript file as config allows arbitrary code execution at
startup — a security concern. Declarative TOML with an explicit `.forge-env`
escape hatch separates configuration from code cleanly.

### Alternative B — Multiple config files like bash

**Rejected:** The fragmented `.bashrc` / `.bash_profile` / `.profile` model
is exactly the complexity we are replacing. One file, one source of truth.

### Alternative C — JSON config

**Rejected:** TOML is more human-readable for configuration files, supports
comments, and is the established convention in the Rust ecosystem (Cargo.toml).

## Unresolved Questions

- [ ] Should project-level `config.toml` support all sections or a restricted
      subset for security?
- [ ] How are config schema versions handled — what happens when a new Forge
      version adds new config keys?
- [ ] Should `forge config set` modify the file in place or append to a
      separate override file?
- [ ] How are conflicting plugin aliases resolved — last-installed-wins or
      explicit precedence declaration?

## Implementation Plan

### Affected Crates

| Crate | Responsibility |
|---|---|
| `forge-config` | Config file parsing, validation, merging |
| `forge-config/platform` | Platform section resolution |
| `forge-config/trust` | Trust grant management |
| `forge-cli/config` | `forge config` CLI subcommands |

### Dependencies

- Requires RFC-004 (path type, env model) — `[path]` and `[env]` sections
- Requires RFC-005 (plugin system) — `[plugins]` section
- Requires RFC-011 (forge migrate) — `forge config migrate` subcommand
- Requires RFC-012 (REPL) — `[repl]`, `[prompt]`, `[keybindings]` sections

### Milestones

1. Define `config.toml` schema — all sections, types, defaults
2. Implement `forge-config` parser and validator
3. Implement platform section resolution — `[env.macos]` etc.
4. Implement config loading order — user → includes → project → `.forge-env`
5. Implement trust model — `forge env trust` and `trusted-envs.toml`
6. Implement `forge config` CLI — `get`, `set`, `list`, `validate`, `init`
7. Implement `forge config migrate` — extract from `.bashrc`/`.zshrc`
8. Integration tests on ubuntu-latest, macos-latest, windows-latest
